## 1. 문서 개요

### 1.1 목적
본 문서는 '노동 법률 종합 AI 어시스턴트' 프로젝트의 기능적/비기능적 요구사항을 정의하여, 기획-화면설계-개발-테스트 전 단계에서 일관된 기준으로 참조할 수 있도록 하는 것을 목적으로 한다.

### 1.2 프로젝트 개요
본 서비스는 대한민국 노동 법률에 대한 질의응답, 권리 분석, 증거 수집 안내 등을 지원하는 AI 어시스턴트로, 법령·판례를 벡터 DB(ChromaDB)에 임베딩하고 LangGraph 기반 멀티스텝 추론(RAG)으로 근거 있는 법률 정보를 제공한다. 4차 프로젝트부터는 Django 프레임워크 기반 웹 애플리케이션으로 재구현되며, Docker 컨테이너화 후 AWS 환경에 배포된다. OpenAI GPT 및 임베딩 API, 네이버 뉴스 API와 연동한다.

### 1.3 용어 정의

| 용어 | 정의 |
|---|---|
| RAG | Retrieval-Augmented Generation. 검색된 문서를 근거로 LLM이 답변을 생성하는 방식 |
| LangGraph | 멀티스텝 추론 파이프라인을 노드/엣지 그래프로 구성하는 LangChain 기반 프레임워크 |
| SAC | Summary-Augmented Chunking. 검색용 요약과 LLM 답변용 원문을 분리하는 임베딩 구조 |
| Router | 사용자 질문을 4개 모드 중 하나로 분류하는 LLM 기반 라우팅 엔진 |
| Supervisor | 복합 질문에 대해 서브에이전트 실행 순서를 결정하는 멀티에이전트 컨트롤러 |
| Django | 4차 프로젝트에서 채택한 Python 웹 프레임워크. 인증, ORM, 템플릿/REST API 등을 제공 |
| Docker | 애플리케이션과 실행 환경을 컨테이너 이미지로 패키징하는 기술. 로컬-스테이징-운영 환경 일관성 확보 목적 |
| AWS | 운영 배포 대상 클라우드 플랫폼(ECS/EC2, S3, RDS, ALB 등) |

### 1.4 문서 범위
- 기능 요구사항: 사용자용 6개 주요 페이지(홈, QA, 권리찾기, 증거자료 관리(증거 수집 안내), 계산기, 최신 뉴스) 및 관리자 콘솔 5개 화면(대시보드, 사용자 관리, 피드백 관리, 프롬프트 관리, 성능 모니터링)
- 비기능 요구사항: 인증·권한, 성능, 반응형 UI, 배포 환경, 보안
- LLM/외부 API 연동 요구사항: OpenAI GPT, OpenAI Embedding, 네이버 뉴스 API
- 사용자 시나리오(유스케이스) 및 화면설계·개발·테스트 추적성 매트릭스

## 2. 기능 요구사항 (Functional Requirements)

### 2.1 인증 및 권한 (Auth & Authorization)

| ID | 요구사항 내용 |
|---|---|
| FR-001 | 사용자는 이메일/비밀번호 기반으로 회원가입 및 로그인할 수 있어야 한다. 비밀번호는 해시(bcrypt 등)로 저장한다. |
| FR-002 | 로그인한 사용자는 Django 세션 프레임워크(session)를 통해 인증 상태를 유지하며, 세션 만료 시간은 60분으로 설정한다. |
| FR-003 | 비로그인 사용자는 로그인/회원가입 페이지 외 모든 기능에 접근할 수 없다. 홈, QA, 권리찾기, 증거자료 관리(증거 수집 안내), 계산기, 최신뉴스를 포함한 모든 기능은 로그인한 사용자만 이용할 수 있다. |
| FR-004 | 사용자는 본인의 QA/권리찾기 질의 이력 및 계산기 사용 이력에 대해서만 조회/삭제 권한(소유자 권한)을 가진다. |
| FR-005 | 관리자 권한을 가진 계정은 프롬프트 템플릿 관리 기능에 접근할 수 있다. |
| FR-006 | 관리자 권한 계정만 관리자 콘솔(대시보드/사용자 관리/피드백 관리)에 접근할 수 있으며, 일반 사용자가 해당 URL에 접근할 경우 403으로 차단한다. |
| FR-008 | 로그인한 사용자는 로그아웃 기능을 통해 세션을 명시적으로 종료할 수 있다. |
| FR-009 | 동일 계정에 대한 로그인 실패가 5회 연속 누적되면 10분간 로그인 시도를 제한하여 무차별 대입(brute-force) 공격을 방지한다. |
| FR-010 | 관리자 계정은 일반 회원가입 화면을 통해 생성되지 않으며, Django manage.py createsuperuser 또는 관리자에 의해 사전 등록된 계정에 한해 관리자 권한(is_staff/is_superuser)이 부여된다. 일반 사용자는 가입 절차만으로 관리자 권한을 획득할 수 없다. |

### 2.2 홈 (대시보드)

| ID | 요구사항 내용 |
|---|---|
| FR-011 | 홈 화면은 서비스 개요를 카드 형태로 노출한다. |
| FR-012 | 로그인한 세션에서는 홈 화면에서 AI 상담, 계산기, 최신 뉴스, 마이페이지 각 기능 페이지로 1-click 이동이 가능해야 한다. |
| FR-013 | 홈 화면의 질의응답 입력창은 비로그인 상태에서는 QA 페이지로 이동하지 않으며, 로그인한 상태에서만 질의응답 기능을 이용할 수 있다. |

### 2.3 QA (법률 질의응답)

| ID | 요구사항 내용 |
|---|---|
| FR-020 | 사용자는 자연어로 노동법 관련 질문을 입력할 수 있다. |
| FR-021 | 시스템은 RouterEngine을 통해 질문을 case_based_answer / procedure_guidance / allowance_calculator / latest_news 4개 모드 중 하나로 분류한다. |
| FR-022 | case_based_answer 모드에서는 LangGraph RAG 파이프라인(판례 검색→법령 검색→답변 생성)을 실행하여 근거 있는 답변을 생성한다. |
| FR-022-1 | procedure_guidance 모드에서는 판례/법령 검색 없이 절차 안내(procedure_guide) 단계만 실행하여 해당 상황에 적용되는 신고·구제 절차를 안내한다. 이 모드는 권리찾기 기능(FR-031)의 procedure_guide_node를 동일하게 활용한다. |
| FR-023 | case_based_answer 모드의 답변에는 참조된 판례(사건번호)와 법령(법령명+조문번호)이 함께 표시되어야 한다. |
| FR-024 | 복합 질문의 경우 SupervisorGraph가 rag_router/calculator/news 서브에이전트를 최대 3회까지 순차 실행하여 통합 답변을 구성한다. |
| FR-025 | 질의 처리 중에는 로딩 상태를 표시한다. |

### 2.4 권리찾기 (상황별 권리 진단)

| ID | 요구사항 내용 |
|---|---|
| FR-030 | 사용자는 근무 상황(예: 임금체불, 부당해고, 직장 내 괴롭힘 등)을 서술형으로 입력할 수 있다. |
| FR-031 | 시스템은 procedure_guide_node를 통해 해당 상황에 적용 가능한 법적 절차(신고, 구제 신청, 증거 확보 등)를 단계별로 안내한다. |
| FR-032 | 진단 결과는 사용자별 이력으로 저장되어 재조회할 수 있어야 한다. |

### 2.5 증거자료 관리 (증거 수집 안내)

| ID | 요구사항 내용 |
|---|---|
| FR-050 | 시스템은 상황(임금체불, 부당해고, 직장 내 괴롭힘 등)별로 확보해야 할 증거자료 목록(예: 근로계약서, 급여명세서, 근태기록, 대화 캡처 등)을 안내 텍스트로 제공한다. |

### 2.6 계산기 (임금·퇴직금·퇴직연금)

사용자는 근로 조건을 숫자로 입력하거나 자연어로 질의할 수 있다.

| ID | 요구사항 내용 |
|---|---|
| FR-060 | 급여, 근무 기간을 입력받으면 예상 퇴직금을 계산한다. |
| FR-061 | 급여를 입력받으면 연차수당을 계산한다. |
| FR-062 | 급여, 근로 시간을 입력받으면 주휴수당을 계산한다. |
| FR-063 | 근로 시간을 입력받으면 최저임금을 위반하는지 확인할 수 있다. |

### 2.8 마이페이지

사용자가 본인의 상담 이력과 계정 정보를 확인할 수 있는 개인 페이지 기능이다.

| ID | 요구사항 내용 |
|---|---|
| FR-095 | 마이페이지는 탭 버튼(Tabs)을 통해 '상담 이력'과 '계정 정보' 화면을 전환할 수 있어야 한다. |

### 2.9 관리자 - 대시보드

관리자 콘솔의 메인 화면으로, 서비스 운영 현황을 한눈에 파악할 수 있는 통계/모니터링 기능이다. (참고: 관리자 대시보드 목업)

| ID | 요구사항 내용 |
|---|---|
| FR-100 | 관리자는 대시보드 상단에서 당일 질문 수, 신규 가입자 수, 평균 만족도, 전체 사용자 수를 전일 대비 증감(%포인트/명)과 함께 카드 형태로 조회할 수 있다. |
| FR-101 | 관리자는 최근 7일간의 일별 질문 수 및 신규 가입 추이를 라인 차트로 조회할 수 있다. |
| FR-102 | 관리자는 임금체불·부당해고·퇴직금·연차휴가·최저임금 등 카테고리별 질문 분포를 도넛 차트와 비율(%)로 조회할 수 있다. |
| FR-103 | 관리자는 카테고리별 질문 증감 추이를 확인할 수 있다. |
| FR-104 | 관리자는 전체 사용자 수와 사용자 활성 상태를 확인할 수 있고, 사용자 관리 화면으로 이동할 수 있다. |
| FR-105 | 관리자는 카테고리별 평가 비율을 확인할 수 있고, 피드백 관리 페이지로 이동할 수 있다. |

| ID | 요구사항 내용 |
|---|---|
| FR-110 | 관리자는 화면 상단에서 전체 사용자 수, 활성 사용자 수, 정지된 사용자 수를 요약 정보로 조회할 수 있다. |
| FR-111 | 관리자는 이름 또는 이메일로 사용자를 검색할 수 있다. |
| FR-112 | 관리자는 사용자 상태(전체/활성/정지)를 드롭다운으로 필터링하여 조회할 수 있다. |
| FR-113 | 사용자 목록에는 사용자명, 이메일, 가입일, 마지막 접속일, 누적 질문 수, 현재 상태가 표시되어야 한다. |
| FR-114 | 관리자는 활성 사용자를 '계정 정지' 처리할 수 있으며, 정지 상태인 사용자는 '정지 해제'를 통해 활성 상태로 되돌릴 수 있다. |
| FR-115 | 관리자는 현재 조회된 사용자 목록을 CSV/Excel 파일로 내보내기(export)할 수 있다. |

### 2.11 관리자 - 피드백 관리

사용자가 QA 답변에 남긴 좋아요/싫어요 평가를 집계하여 답변 품질을 모니터링하고 개선 이력을 관리하는 기능이다. (참고: 피드백 관리 목업)

| ID | 요구사항 내용 |
|---|---|
| FR-120 | 관리자는 전체 좋아요 수, 전체 싫어요 수, 평균 만족도(%)를 조회할 수 있다. |
| FR-121 | 관리자는 카테고리별 만족도를 오름차순으로 확인할 수 있다. |
| FR-122 | 관리자는 전체 카테고리와 개선 필요 탭을 눌러 화면을 이동할 수 있다. |
| FR-123 | 관리자는 카테고리의 아코디언을 누르면 일자별 총 좋아요 수와 싫어요 수를 확인할 수 있다. |
| FR-124 | 만족도가 60% 미만인 답변에는 '개선 필요' 배지가 자동으로 표시된다. |

### 2.12 관리자 - 프롬프트 관리

법률 답변 품질에 직접 영향을 미치는 프롬프트 템플릿(backend/prompts/*.md 5종)을 관리자가 안전하게 수정·검증·롤백할 수 있도록 하는 기능이다.

| ID | 요구사항 내용 |

- README.md에는 인증·권한 관련 구현이 명시되어 있지 않아, 본 문서의 FR-001~010 및 NFR-005~007, NFR-015는 요구사항 정의 목적의 제안 사항으로 별도 확정이 필요하다.
